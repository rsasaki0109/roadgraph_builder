# Memory profile — roadgraph_builder v0.7.0

Input: `examples/osm_public_trackpoints.csv` (Paris OSM public GPS trackpoints, 800 rows)

Measurements taken on: Linux 6.14 / Python 3.12 / Intel Core i7 (dev machine).
Each run is an isolated subprocess; RSS is read via `resource.getrusage(RUSAGE_SELF).ru_maxrss`.

## Before / After: peak RSS (export-bundle, lane_width_m=3.5)

| Measurement | Peak RSS (KB) | Peak RSS (MB) |
| ----------- | ------------- | ------------- |
| **Before** (v0.7.0 pre-V3, minidom round-trip) | 61028 | 59.6 |
| **After** (V3 optimized, direct XML writer) | 54944 | 53.7 |
| **Reduction** | **6084 KB** | **10.0 %** |

tracemalloc peak (post-bundle vs import baseline): ~10 MB.

## Hotspot identified: minidom round-trip in `export_lanelet2`

The biggest non-import allocator was the XML DOM parse in
`roadgraph_builder/io/export/lanelet2.py`:

```python
# BEFORE (expensive path):
xml_bytes = ET.tostring(root, encoding="utf-8")
dom = minidom.parseString(xml_bytes)          # ~900 KB of DOM objects
pretty = dom.toprettyxml(indent="  ", encoding="utf-8")
path.write_bytes(pretty)
```

`minidom.parseString` allocated ~900 KB of DOM node objects
(~4 200 × minidom elements for a Paris-scale graph with lanelets and
regulatory_element relations). These objects were discarded immediately
after `toprettyxml`.

## Fix applied: `_et_to_pretty_bytes()` direct serializer

Replaced with a recursive byte-chunk writer that produces byte-identical
output (no space before `/>`, same indentation, same XML declaration):

```python
# AFTER (optimized path):
path.write_bytes(_et_to_pretty_bytes(root))
```

`_et_to_pretty_bytes` appends `bytes` chunks to a list and joins once;
no intermediate DOM is created.  Output is verified byte-identical to
the minidom path on the Paris real-data graph.

allocators become dominant and the relative benefit of the minidom fix grows
proportionally with edge count.

## Remaining hotspots (FOLLOWUP)

- **`crossing_splitters.py:288` — grid cell tuples** (`(c, r)` tuple per cell,
  ~70 KB for Paris): switching the grid dict key to a packed integer
  `c * MAX_R + r` would halve the tuple overhead.  Safe but deferred.
- **`trajectory xy float64 → float32`**: now available as an opt-in prototype
  through `load_trajectory_csv(..., xy_dtype="float32")`,
  `BuildParams(trajectory_xy_dtype="float32")`, CLI `--trajectory-dtype
  float32`, and `scripts/profile_memory.py --trajectory-dtype float32`.
  Measurements are in [`float32_drift_report.md`](./float32_drift_report.md):
  Berlin 7,500 rows saved 60,000 B in the loader allocation and kept topology
  unchanged with <1 mm coordinate drift. A 1M-row synthetic run saved exactly
  8,000,000 B in the retained trajectory XY allocation and dropped
  tracemalloc peak by ~19 MB, but process-level RSS after full export only
  fell by ~2.6 MB because build/export temporaries dominate. Default remains
  float64.
- **Polyline list-of-tuples → NumPy array**: saves ~80 % on Python object
  overhead for large graphs (1 000 polylines × 20 pts: 1 336 KB → 312 KB).
  Requires updating all consumers of `edge.polyline`; complex refactor.
